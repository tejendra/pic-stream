# 2.1 Define Firestore collections and indexes – Implementation plan

## Summary

Define Firestore collection `albums` and subcollection `albums/{albumId}/media` with specified fields, and add composite indexes for querying by `deleteOn` (albums) and by `albumId` + `createdAt` (media).

## Approach

- **Backend**: Firestore does not require explicit collection creation; collections appear when the first document is written. We define the schema in this plan and in `firestore.indexes.json` so that indexes exist and future writes (from album/media APIs) use the correct field names. No new backend code is required for “defining” collections; types and write paths are added in work items 2.3 and 3.x.
- **Indexes**: Add to `firestore.indexes.json`:
  1. **albums by deleteOn**: Firestore provides a built-in single-field index for `deleteOn`; no explicit composite is required (Firebase rejects single-field entries in `indexes`). Queries by `deleteOn` work without adding an index.
  2. **media**: composite index on `albumId` ascending + `createdAt` ascending. Media subcollection documents include an `albumId` field (denormalized from the path) so collection-group or scoped queries can use this index.
- **Schema (reference)**:
  - **albums**: `id`, `name`, `seedHash`, `deleteOn`, `createdAt`, `createdBy`.
  - **albums/{albumId}/media**: `id`, `albumId`, `storagePath`, `previewPath`, `thumbnailPath`, `displayName`, `uploaderName`, `size`, `mimeType`, `duplicateKey`, `createdAt`.
