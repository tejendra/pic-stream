# How Firebase is used in Pic Stream

This document describes which Firebase products the project uses and how they fit into the architecture.

## Overview

| Product       | Purpose |
|--------------|---------|
| **Firestore** | Database for albums and media metadata. |
| **Storage**   | File storage for photos and videos (originals, thumbnails, previews). |
| **Hosting**   | Serves the built React SPA (static files from `frontend/dist`). |

The frontend does **not** use any Firebase SDK. All Firestore and Storage access goes through the Express API, which uses the **Firebase Admin SDK** on the server.

## Architecture

```
┌─────────────┐      ┌─────────────────┐      ┌──────────────────┐
│   Browser   │ ──── │  Express API    │ ──── │  Firebase Admin  │
│  (React)    │      │  (backend/)     │      │  Firestore +      │
│  No Firebase│      │  firebase-admin │      │  Storage          │
└─────────────┘      └─────────────────┘      └──────────────────┘
       │                       │
       │                       │  (Hosting serves SPA only;
       └───────────────────────┼   no direct Firebase from client)
                               │
                     ┌─────────▼─────────┐
                     │  Firebase Hosting │  ← static files only
                     │  (frontend/dist)  │
                     └──────────────────┘
```

- **Frontend:** Calls the backend at `VITE_API_URL` (or same origin in dev via proxy). No Firebase client SDK.
- **Backend:** Uses `firebase-admin` to read/write Firestore and to generate signed upload/download URLs for Storage. Runs on Cloud Run (or locally with `GOOGLE_APPLICATION_CREDENTIALS`).

## Firestore

- **Used for:** Albums and media metadata (planned: `albums` collection, `albums/{id}/media` subcollection).
- **Backend usage:** `backend/src/lib/firestore.ts` – lazy init via Admin SDK, then `readDoc` / `writeDoc` and direct Firestore access from services.
- **Credentials:** Application Default Credentials (env `GOOGLE_APPLICATION_CREDENTIALS` locally; automatic on Cloud Run).
- **Rules:** `firestore.rules` – evaluated only for **client** access. The Admin SDK bypasses rules; rules are for defense in depth (e.g. deny all client access so only the backend can use Firestore).

## Storage

- **Used for:** Storing photo and video files (originals, thumbnails, previews). The backend never streams file bytes through the server; it issues **signed URLs** so the client uploads/downloads directly to/from Storage.
- **Backend usage:** `backend/src/lib/storage.ts` – `getSignedUploadUrl(path)` and `getSignedDownloadUrl(path)` (v4, configurable expiry).
- **Bucket:** Set via `FIREBASE_STORAGE_BUCKET` in backend env.
- **Rules:** `storage.rules` – deny all client read/write (`if false`). All access is via signed URLs generated by the backend; rules block direct client access to the bucket.

## Hosting

- **Used for:** Serving the built React app (SPA).
- **Config:** `firebase.json` → `hosting.public` = `frontend/dist`. A rewrite sends all routes to `/index.html` for client-side routing.
- **Deploy:** Build the frontend (`cd frontend && npm run build`), then `firebase deploy --only hosting` (or `firebase deploy` to update rules and hosting together).

## Configuration

Firebase is only initialized when both are set:

- `GOOGLE_CLOUD_PROJECT` – Firebase / Google Cloud project ID.
- `FIREBASE_STORAGE_BUCKET` – Storage bucket (e.g. `your-project.appspot.com`).

Optional for local dev:

- `GOOGLE_APPLICATION_CREDENTIALS` – Path to a service account JSON key. Omit on Cloud Run (default credentials are used).

Initialization is **lazy**: the backend starts even if these are unset; Firestore/Storage are only used when configured and when a request needs them. See `backend/src/lib/firebase.ts` and `backend/src/config.ts`.

## What we don’t use

- **Firebase client SDK** in the frontend – the app talks only to the Express API.
- **Data Connect** – not used; data access is via the REST API and Admin SDK.
- **Cloud Functions** – the API is an Express app deployed to **Cloud Run**, not Functions.
- **App Hosting** – the backend is Express on Cloud Run; the frontend is static files on Firebase Hosting.

## Summary

Firebase in Pic Stream = **Firestore** (data) + **Storage** (files) + **Hosting** (SPA). Only the backend touches Firestore and Storage, using the Admin SDK. The frontend uses no Firebase services directly.
